# Анализ проекта pyMailcowAliasesBot

## 📋 Общее описание

**Назначение**: Telegram-бот для создания email-алиасов на почтовом сервере mailcow. Позволяет пользователям легко создавать alias'ы вида `site@example.com` без доступа к панели управления mailcow.

**Стек**: Python 3.11+, telebot, requests, sqlite3

**Версия**: 0.1.1

---

tg_mailcow_aliases/

## 🔍 Детальный разбор компонентов

### 1. `bot.py` - Основной Telegram-бот

**Основной функционал**:
- Инициализирует telebot с токеном из переменной окружения `TG_BOT_TOKEN`
- Обрабатывает команды `/start` и `/help`
- Обрабатывает все текстовые сообщения как название alias'а

**Команды**:
- `/start`, `/help` - показывают приветственное сообщение с информацией пользователя в формате:
  ```
  Hi {name}, aliases for {domain} is goto {goto}
  ```

**Основной функционал - создание alias'а**:

```python
@bot.message_handler(func=lambda message: True)
def handle_message(message):
    # Пользователь отправляет текст (например, "testsite")
    # Бот проверяет наличие пользователя в БД
    # Получает домен и адрес для перенаправления из БД
    # Создает alias "testsite@domain.com → email@example.com"
    # Отправляет результат пользователю
```

**Логика работы**:
1. Получает текст сообщения как название alias'а (левая часть)
2. Проверяет наличие пользователя в БД (возвращает 403 Forbidden если нет)
3. Извлекает конфигурацию пользователя (домен и адрес для перенаправления)
4. Вызывает `add_alias()` для создания на mailcow
5. Парсит ответ сервера:
   - **Success**: отправляет ✅ сообщение с адресом и целевым email
   - **Error**: отправляет 🆘 сообщение с типом ошибки и описанием

**Пример ответа от mailcow при успехе**:
```json
[{
  "log": [
    "mailbox",
    "add",
    "alias",
    {
      "active": "1",
      "address": "testsite@example.com",
      "goto": "destination@example.com"
    },
    null
  ],
  "msg": ["alias_added", "testsite@example.com"],
  "type": "success"
}]
```

### 2. `api.py` - Интеграция с API mailcow

**Конфигурация** (из переменных окружения):
- `SERVER_URL` - базовый URL mailcow сервера
- `X_API_KEY` - ключ API для аутентификации
- `DEFAULT_DOMAIN` - домен по умолчанию
- `DEFAULT_EMAIL` - email по умолчанию

**API endpoints**:
```python
GET_ALIAS = "/api/v1/get/alias"
ADD_ALIAS = "/api/v1/add/alias"
DELETE_ALIAS = "/api/v1/delete/alias"
```

**Функции**:

#### `add_alias(new_alias, goto, domain, active=1)`
- Создает новый alias на mailcow сервере
- **Параметры**:
  - `new_alias` - левая часть alias'а (например, "testsite")
  - `goto` - целевой email для перенаправления
  - `domain` - домен для alias'а (правая часть)
  - `active` - статус активности (по умолчанию 1)
- **Возвращает**: кортеж `(status_code, response_json)`
- **Пример**: `add_alias("testsite", "user@example.com", "mydomain.com")`

#### `delete_alias(alias_id)`
- Удаляет alias по ID
- **Параметры**: `alias_id` - ID alias'а в системе mailcow
- **Возвращает**: кортеж `(status_code, response_json)`

**Заголовки запросов**:
```python
HEADER = {
    "Accept": "application/json",
    "Content-Type": "application/json",
    "X-API-Key": X_API_KEY,
}
```

### 3. `sqlite.py` - Управление пользователями

**Структура таблицы `users`**:
```sql
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id TEXT,              -- ID пользователя в Telegram
    user_name TEXT,            -- Имя/ник
    domain TEXT,               -- Домен для создания alias'ов
    goto TEXT                  -- Email для перенаправления
)
```

**CRUD операции**:

#### `create_tabel(db=DB_PATH)`
- Создает таблицу users если её еще нет
- **Параметры**: `db` - путь к БД (по умолчанию `/data/user.db`)
- **⚠️ Опечатка**: функция названа `tabel` вместо `table`

#### `add_user(id, name, domain, goto, db=DB_PATH)`
- Добавляет нового пользователя в БД
- **Параметры**:
  - `id` - Telegram ID пользователя
  - `name` - Имя/никнейм
  - `domain` - Домен для alias'ов
  - `goto` - Email для перенаправления

#### `is_user_exist(id, db=DB_PATH)`
- Проверяет наличие пользователя в БД
- **Возвращает**: `True` если пользователь есть, иначе `False`

#### `get_user(id, db=DB_PATH)`
- Получает данные пользователя
- **Возвращает**: кортеж `(user_id, user_name, domain, goto)` или `None`

#### `delete_user(id, db=DB_PATH)`
- Удаляет пользователя из БД
- **Параметры**: `id` - Telegram ID пользователя

#### `get_all_users(db=DB_PATH)`
- Получает список всех пользователей
- **Возвращает**: список кортежей со всеми полями

## ⚠️ Проблемы и замечания

### Критические

#### 1. ❌ Отсутствие error handling для сетевых ошибок (`api.py`)
- Если mailcow недоступен, произойдет необработанное исключение
- `requests.post()` может выбросить `ConnectionError`, `Timeout` и т.д.
- **Решение**: добавить try-except блоки в `add_alias()` и `delete_alias()`

#### 2. ❌ Отсутствие валидации входных данных
- Нет проверки формата email в `api.py`
- Нет валидации alias'а перед отправкой на сервер
- **Решение**: добавить regex валидацию или использовать `email-validator`

#### 3. ❌ Отсутствие type hints
- Код сложнее тестировать и поддерживать
- IDE не может подсказать типы параметров
- **Решение**: добавить type hints для всех функций

### Высокий приоритет

#### 4. ⚠️ Опечатка в названии функции
- `create_tabel()` вместо `create_table()`
- Используется везде в коде, поэтому требует осторожности при исправлении
- Может путать при поддержке кода

#### 5. ⚠️ Hardcoded пути
- `/data/user.db` в `sqlite.py`
- `/data/test_users.db` в тестах
- **Проблема**: сложно тестировать локально, нет гибкости конфигурации
- **Решение**: использовать переменные окружения или параметры функций

#### 6. ⚠️ Недостаточное логирование в `bot.py`
- Логируются ID и имена, но нет логирования полной цепочки ошибок
- Сложно дебажить проблемы при создании alias'а
- **Решение**: добавить логирование ошибок от mailcow API

#### 7. ⚠️ Отсутствие обработки исключений в БД операциях (`sqlite.py`)
- Нет try-except блоков при работе с sqlite
- Если произойдет ошибка БД, приложение упадет
- **Решение**: обернуть все операции в try-except

### Средний приоритет

#### 8. 📝 Строковые ID пользователей
- ID конвертируется в строку без причины в `bot.py` (line 59):
  ```python
  user_id = str(message.from_user.id)  # Зачем конвертировать?
  ```
- Но в `is_user_exist()` передается как строка, а в `get_user()` тоже
- **Решение**: использовать везде строки или везде интегеры (строки безопаснее для SQL)

#### 9. 📝 Нет проверки статуса при создании БД
- `create_tabel()` не проверяет, успешно ли создалась таблица
- Нет возврата статуса успеха/ошибки
- **Решение**: добавить проверку и логирование

#### 10. 📝 Дублирование кода
- SQL для создания таблицы написан в двух местах:
  - `tg_mailcow_aliases/sqlite.py`
  - `tests/test_sqlite.py`
- **Решение**: вынести в одно место, использовать везде

#### 11. 📝 Нет кэширования данных пользователя
- При каждом сообщении делается запрос в БД
- Для high-load сценариев это может быть узким местом
- **Решение**: добавить простое кэширование в памяти

#### 12. 📝 Отсутствие rate limiting
- Пользователь может спамить создание alias'ов
- Нет защиты от DOS атак
- **Решение**: добавить rate limiting в бот
